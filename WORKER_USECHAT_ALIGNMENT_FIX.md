# Worker / useChat 对齐修复说明

## 背景

本次修复目标：解决 worker 模式与 useChat 模式在消息解析与入库结构上的关键不一致问题。

## 修复的问题

### 问题 1：Stage 文本格式不一致，导致前端识别差异

- 现象：worker 侧阶段信息文本与 useChat 侧不一致（如冒号、空格格式差异），会影响前端对阶段消息的正则识别和分组展示。

### 问题 6：SSE 流结束时尾包解析不完整，可能丢消息

- 现象：在 `done` 分支仅对剩余 buffer 做一次解析，若尾部包含多条 `data:` 事件，可能只消费一部分，导致最后几条消息丢失或未入库。

## 根因分析

### 问题 1 根因

1. worker 组装 stage 文本时的格式未严格与 useChat 对齐。
2. stage 解析后可能残留前导 `:`，导致拼接结果在不同来源下表现不一致。
3. 前端正则对格式较严格，历史/异常格式兼容性不足。

### 问题 6 根因

1. SSE 尾包处理按“整段一次性解析”设计，未按事件粒度逐行处理。
2. SSE 天然是多事件流，最后一次 `reader.read()` 完成时，buffer 可能残留多条 `data:` 行。

## 修复方式

### 对问题 1 的修复

1. worker 统一 stage 输出格式为：`[STAGE]: message`。
2. 在 stage 解析阶段，去除 message 前导 `:`（如 `": xxx"` -> `"xxx"`），防止重复冒号或脏格式传播。
3. 前端 stage 正则增强兼容性，允许有/无冒号格式，兼容历史数据。

### 对问题 6 的修复

1. 在 `done` 分支中，将尾部 buffer 按行拆分。
2. 逐条处理 `data:` 行并独立 JSON 解析。
3. 每条事件都走现有分类与缓存合并逻辑，避免尾部多事件丢失。

## 影响范围

- worker 消息解析与输出：`src/workers/chatStreamingWorker.ts`
- 前端 stage 识别：`src/app/studio/basicUI.tsx`

## 结果

- worker 与 useChat 在 stage 消息格式上的一致性提升。
- SSE 结束阶段尾包多事件可被完整处理，降低丢消息风险。
- 历史数据展示兼容性增强。
